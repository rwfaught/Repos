# First Real Non-Sensitive Dossier Engagement Operator Packet

## 1. Status and Authority

**Packet status:** preparation artifact only.

- Engagement execution is not yet authorized.
- Participant contact is not yet authorized.
- Storage surfaces are not generally approved.
- This is not a client-ready runtime claim.
- This is not a legal, compliance, privacy, or security claim.
- CTO/coordinator must separately authorize participant contact and execution.

## 2. Engagement Definition

**Only eligible format:** one authorized participant, one narrow workflow,
verbal abstraction, recommendation-only, 30–45 minutes, no implementation, and
no raw material.

**Intended output:** a brief workflow summary; 5–12 process steps; friction
categories; contradictions; missing information; constraints; 1–2 preliminary
opportunities; limitations; and one bounded next question.

**Automatic exclusions:** personal, customer, employee, patient, financial,
payment, credential, proprietary-record, legal, insurance, benefits, safety,
emergency, hiring/firing, eligibility, credit, pricing-decision, or other
regulated/consequential material; production access; recording; implementation;
or any sensitivity that cannot be safely abstracted.

## 4. Operational Definition of Non-Sensitive

| Category | Examples | Action |
| --- | --- | --- |
| Acceptable process-level information | “A coordinator checks a request, then a technician reviews it”; approximate volume ranges; role labels; generic delay categories. | Note only in abstract form. |
| Abstraction-required information | “A customer name appears in the request”; a particular software product; an exact dollar amount; an identifiable incident. | Replace with role, category, or range before noting. |
| Prohibited information | Names, contacts, account numbers, credentials, screenshots, files, recordings, health/financial details, customer records, exact proprietary documents. | Do not receive, repeat, or retain it. Stop if it enters the session. |
| Unresolved sensitivity posture | The participant cannot tell whether a detail is restricted, or the operator is unsure. | Stop and return to CTO/coordinator. |

`UNCERTAINTY_MEANS_STOP_NOT_PERMISSION`

## 5. Pre-Engagement Disclosure Script

“This is a short, limited discovery conversation about one workflow. We will
use only roles, categories, ranges, and fictional examples—please do not share
names, records, screenshots, account details, credentials, or restricted
information. I will not record the conversation, access any system, or
implement anything. I will make only temporary minimal paper notes and can
pause or stop at any time. The outcome, if we continue, is a recommendation-
only summary, not compliance, legal, or professional advice. Do you have
authority to discuss this workflow at this abstract level, and do you consent
to these limits and stop rules?”

## 6. Session Setup Checklist

- [ ] Assign generic engagement code: `ENG-____`.
- [ ] Prepare paper only; write no business or participant name.
- [ ] Confirm no active recording.
- [ ] Do not open a digital engagement notebook.
- [ ] Do not open a model, RAG, Orchestrator, or other system surface for data entry.
- [ ] Prepare stop phrase: “I need to stop here; that detail is outside this session.”
- [ ] Set a 30–45 minute timer.
- [ ] Write one-sentence scope: “Abstract discussion of [one workflow] only.”

## 7. Interview Guide

Ask for abstractions, never examples containing real records.

1. What general event triggers this workflow?
2. Which roles participate, without naming people?
3. What are the main 5–12 steps from trigger to completion?
4. How does information move between roles or tools, in categories only?
5. Where is information copied, re-entered, or summarized repeatedly?
6. Where do people wait, and what category causes the delay?
7. What information is commonly missing?
8. What kinds of statements or instructions sometimes conflict?
9. Which human approvals must remain explicit?
10. What normally happens when the workflow fails or stalls?
11. What would “better” look like in observable, non-sensitive terms?
12. What must never be automated or delegated in this workflow?

## 8. Mid-Session Abstraction Checkpoint

1. Pause.
2. Summarize notes using roles, categories, and ranges.
3. Ask whether restricted detail entered the notes.
4. Ask whether anything should be removed.
5. Confirm the remaining abstracted material is accurate.
6. Stop if safe abstraction cannot continue.

## 9. Note Classification Card

| Label | Definition | Example | Required action |
| --- | --- | --- | --- |
| `SAFE_NOTE` | Process-level abstraction with no identifier or restricted content. | “Two role handoffs; one approval.” | Retain temporarily on paper. |
| `REDACT_IMMEDIATELY` | A safe note contaminated by a removable sensitive detail. | “Customer [name] waits two days.” | Strike out the unsafe detail; retain only “requester waits 1–3 days.” |
| `PROHIBITED_NOTE` | Restricted content that must not be retained. | Account number, name, credential, document detail. | Stop, do not copy, and destroy the unsafe original. |
| `SENSITIVITY_UNRESOLVED` | Safety classification is uncertain. | “This internal status may be confidential.” | Stop; do not retain or continue; return to CTO/coordinator. |

Destroy the unsafe original rather than storing it beside a redacted version.

| Label | Use |

| `PARTICIPANT_CLAIM` | Participant’s unverified description. |

| `APPROVAL_TO_DISCUSS_FURTHER` | Permission for a later discussion only, if separately authorized. |
| `IMPLEMENTATION_AUTHORIZATION` | Explicit authorization for implementation. Not available here. |

No implementation authorization can arise inside this engagement. No statement
may be promoted to a higher authorization level by interpretation.
Recommendation is not authorization. Participant interest is not
willingness-to-pay proof.

## 11. Immediate Stop Card

Say: **“I need to stop here; that is outside the approved abstract discussion.”**

| Trigger | Immediate action |
| --- | --- |
| Sensitive information | Stop the detail; do not note or repeat it. |
| Production-access request | Decline; do not access any system. |
| Regulated or consequential decision | Stop the engagement. |
| Unclear consent | Pause; do not proceed without separate clarification. |
| Recording request | Decline; stop if recording remains requested. |
| Implementation request | State that implementation is not authorized. |
| Unsafe workflow description | Stop when abstraction cannot make it safe. |
| Restricted data entered an unapproved system | Follow the incident posture below. |

**Unapproved-system incident posture:**

1. Stop.
2. Stop further copying or synchronization.
3. Do not reproduce the restricted content.
4. Identify the category and affected system only.
5. Return to CTO/coordinator.
6. Do not claim deletion or remediation succeeded.
7. Do not resume.

## 12. Temporary Paper-Note Protocol

- Use the generic engagement code only; no names or identifiers.
- Capture process-level notes only.
- Mark every sheet: `TEMPORARY — DESTROY AFTER CLOSEOUT`.
- Keep the paper under Roger’s physical control.
- Do not photograph or scan it.
- Digital substantive engagement notes remain unauthorized.
- After participant review and de-identified evidence extraction, destroy the notes.
- Retain no longer than seven calendar days.
- Record destruction completion only as a yes/no marker.
