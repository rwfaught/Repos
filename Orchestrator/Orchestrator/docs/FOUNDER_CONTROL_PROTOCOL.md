# Founder Control Protocol

Boundary: `FOUNDER_CONTROL_AND_SOURCE_REALITY_DOCS_ONLY`

Status: docs-only founder visibility protocol.

Marker: `FOUNDER_CONTROL_AND_SOURCE_REALITY_DOCS_ONLY_REGISTERED=DOCS_ONLY`

## Executive Summary

This protocol exists to keep Orchestrator founder-legible while the project
continues to accumulate machinery. It is not a slowdown mechanism and it is not
a replacement for implementation. It is a cockpit discipline: at important
hinges, the project must show Roger what exists, what is proven, what is only
documented, what strategic direction is being implied, and what decision still
requires founder ratification.

The protocol is triggered when the project risks moving from governed momentum
into black-box momentum.

## Why This Exists

Orchestrator is meant to govern AI-assisted work, not bury the Founder under
AI-generated process. A large amount of governance, readback, phase, and packet
machinery can be valuable only if Roger can still understand the project body:
what it can actually do, which parts are scaffolding, and which product choices
are being silently locked in.

This protocol preserves a simple rule:

Roger can delegate implementation, but he must not accidentally delegate product
destiny.

## Trigger Conditions

A Founder Comprehension Snapshot or equivalent source-reality check is required
before ranking further implementation when any of the following occurs:

- Roger reports black-box concern.
- A Codex or worker campaign has completed.
- A multi-phase campaign has changed the source/control surface.
- A major handoff would otherwise become the source of continuity.
- A product wedge, proving domain, or first user-facing surface is being changed.
- A product wedge is being reinforced without fresh founder ratification.
- A new implementation campaign is about to begin.
- Another governance/readback phase is being ranked instead of visible product
  proof.
- The latest recommendation depends on prior PASS markers that do not prove the
  present strategic question.

## Founder Ratification Gates

The CTO/coordinator may rank operational moves. It may not decide the following
on Roger's behalf:

- first product wedge
- first proving domain
- product identity
- customer/user priority
- acceptable autonomy threshold for real user work
- whether claims/disputes/appeals remains the first proving ground
- whether game/worldbuilding/design dossier work becomes the first proving
  ground
- whether a neutral dossier/case abstraction should precede domain selection

When one of those decisions is implied, the correct output is a ratification
surface, not another implementation phase.

## Required Distinctions

Every founder-facing hinge report must separate:

- accepted facts from inference
- deterministic proof from semantic proof
- source/test/docs behavior from runtime/provider/model behavior
- builder machinery from product capability
- governance/readback artifacts from user-facing artifacts
- parked technical threads from authorized next moves
- historical product strategy from current founder ratification

## Phase 387 Parking Rule

Phase 386 preserved Phase 387 as a previously recommended technical boundary:

`PHASE387_PRODUCT_TASK_PACKET_HANDOFF_PACKET_ESCALATION_OUTCOME_REVIEW_POSTURE_REVIEW_READBACK_SOURCE_TEST_DOCS`

Phase 386 also recorded:

`CAMPAIGN_CAP_REACHED_NO_PHASE_387_AUTHORIZED`

Therefore Phase 387 remains a parked technical thread. It must not resume merely
because it is the next numbered phase. It can resume only after founder
visibility is restored and Roger authorizes that line of work.

## Product Wedge Reopening Rule

Claims/disputes/appeals-style casework exists in the current strategy/design
stack as the previously selected proving domain. That historical record remains
important, but it is not current founder ratification by itself.

Until Roger explicitly ratifies or replaces it, claims/disputes/appeals must be
treated as:

- historically documented
- source-relevant
- not discarded
- not currently allowed to dominate product direction by inertia

## Option C Strategic Bridge

Roger authorized the interim strategic bridge known as Option C:

Create or preserve a neutral dossier/case abstraction before forcing a final
choice between claims/disputes/appeals and game/worldbuilding/design dossier
work.

Option C means:

- do not throw away case-packet work
- stop treating claims/disputes/appeals as the gravitational center until Roger
  ratifies it again
- preserve the structural learnings from case packets
- evaluate whether a more general dossier packet can cover both consequential
  admin casework and creative/worldbuilding design work
- keep the first product wedge decision explicit rather than inherited

Option C is not final wedge selection. It is a bridge that prevents premature
identity lock-in.

## White-Paper Style Standard

Founder-facing overview docs should be written as project-manager readable
white-paper notes:

- start with an executive summary
- use plain English before internal terminology
- define terms at first use
- keep caveats visible but not buried
- avoid long ritual phase prose when a shorter cockpit summary is better
- record only what a founder/operator needs to understand the project as it
  unfolds
- push implementation detail into appendices or source references

The goal is not shallow simplification. The goal is load-bearing clarity.

## Context Saturation Rule

When a session becomes long enough that founder-alignment details may be lost,
the CTO/coordinator should say so directly and recommend exporting the live
understanding into durable docs or a handoff.

A context export should preserve:

- current founder questions
- terms clarified in the session
- decisions Roger explicitly made
- decisions still unratified
- current source reality
- parked technical threads
- next recommended boundary

## Non-Proofs

This protocol does not implement product behavior, execute providers/models,
run workers, refresh Source Files, prove semantic correctness, prove production
readiness, authorize Phase 387, or select the first product wedge.
