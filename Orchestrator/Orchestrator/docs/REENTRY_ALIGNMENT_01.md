# REENTRY_ALIGNMENT_01.md

## Re-entry Alignment 01: Integrate Restart Discipline Into Governance Stack

---

## Goal

Integrate `REENTRY_PROTOCOL_01.md` into the project’s active governance/orientation stack through small, explicit documentation updates.

This is a **docs/control-surface alignment artifact**.

It is not:
- a feature phase
- a fix phase
- a product-strategy artifact
- a code implementation task

Its purpose is narrower:

to make the new re-entry protocol part of the repo’s actual startup and orchestration discipline rather than leaving it as an isolated document.

---

## Why This Artifact Exists

`REENTRY_PROTOCOL_01.md` has been ratified as the project’s restart discipline.

That means it should influence how future sessions and future Orchestrators regain orientation after interruption.

If the rest of the governance stack does not reference it, the protocol will exist only as a passive document instead of an active operating rule.

This artifact exists to close that gap in a bounded way.

---

## Scope

Add narrow references or short alignment passages to the minimum set of docs that should explicitly inherit the new re-entry protocol.

Preferred target set:

- `docs/STARTUP_BRIEF.md`
- `docs/ORCHESTRATOR_METHOD.md`
- `docs/PROJECT_CONTEXT.md`

Optional only if clearly useful:
- another governance/startup doc already responsible for session initialization or handoff

Choose the smallest set of edits that makes re-entry discipline part of the live governance stack.

---

## This Artifact Is NOT About

- broad doc rewriting
- changing project strategy
- changing phase history
- changing product behavior
- adding code
- duplicating the full contents of `REENTRY_PROTOCOL_01.md`
- creating a new startup system from scratch

The goal is alignment, not diffusion.

---

## Required Alignment Outcomes

### 1. Startup alignment
`STARTUP_BRIEF.md` should acknowledge that restart discipline is now governed by `REENTRY_PROTOCOL_01.md`.

This should remain small.
The startup read set should not be rewritten wholesale unless necessary.

### 2. Method-level alignment
`ORCHESTRATOR_METHOD.md` should reflect that:
- repo truth governs restart
- docs-first re-entry is expected
- fresh snapshots or targeted files are requested when current code state matters

This should reinforce, not replace, existing ranking and governance rules.

### 3. Context-level alignment
`PROJECT_CONTEXT.md` should acknowledge that restart/orientation now relies on:
- the governance stack
- the phase ledger
- targeted fresh code evidence when required

This should preserve the distinction between:
- repo truth
- conversational continuity

---

## Design Rules

1. Keep edits small.
2. Prefer short references over long restatements.
3. Do not duplicate the full protocol text.
4. Preserve each document’s existing purpose.
5. Keep restart discipline answerable to repo truth.
6. Do not broaden into a general documentation reorganization.

---

## Source Of Truth

For restart discipline:
- `docs/REENTRY_PROTOCOL_01.md`

For present project truth:
- `docs/PHASE_INDEX.md`
- `docs/ACTION_LOG.md`

For orchestration/governance behavior:
- `docs/ORCHESTRATOR_METHOD.md`

For startup orientation:
- `docs/STARTUP_BRIEF.md`

This artifact should strengthen those relationships without blurring them.

---

## Validation Requirements

Confirm that after the edits:

1. `REENTRY_PROTOCOL_01.md` is referenced where needed in the governance stack
2. startup/orientation docs now reflect docs-first, repo-truth restart discipline
3. no document was broadly rewritten
4. no code or product behavior was changed
5. the alignment remains small and legible

---

1. Apply the bounded alignment edits
2. Update `docs/ACTION_LOG.md`
3. Stop immediately after completion

When done, report:
