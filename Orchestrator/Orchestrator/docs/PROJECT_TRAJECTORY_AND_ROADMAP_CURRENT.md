## 1. Purpose, Scope, and Authority

This document is the current management synthesis for Orchestrator's project
stage, strategic sequence, roadmap phases, entry and exit gates, bottlenecks,
and priorities. It answers the Project Manager-level question: where are we on
the project journey, and what must be true before the project advances?

It is authoritative only for management-level project-stage interpretation,
strategic sequencing, milestone structure, roadmap phase definitions, entry
and exit gates, current strategic priorities, and management-level bottlenecks
and risks.

It is not authoritative for live Git state, source behavior, test results,
runtime proof, implementation details, active open-thread granularity, role
behavior, mutation authorization, product-wedge selection, provider/model
selection, or production readiness. The recorded SHA is this revision's
evidence basis, not perpetual current Git authority.

## 2. Executive Project Position

Orchestrator is a trustworthy general governing layer for AI-assisted work. It
is in a post-foundation, pre-productization alpha stage: the canonical bounded
execution spine is real and authoritative, one exact provider-backed task has
been proven and closed, and the case/dossier substrate is early but real.

The Small-Business AI Implementation Dossier is the first revenue-bearing
proving use case. It is an evidence-generating consulting workflow, not the
Orchestrator product or a selected first product wedge. No first product wedge
has been selected, no architectural lock-in is authorized, and production readiness is not achieved.

Governance strength is substantial. The primary strategic challenge is turning
that discipline into useful, founder-legible work without allowing framework or
readback expansion to substitute for a concrete capability need.

## 3. Vision and Strategic Thesis

The constitutional direction is to extend disciplined, human-legible control of
bounded AI-assisted work outward from execution toward useful objective intake
and supervised workflows. Growth must preserve boundedness, inspectability,
explicit state, reviewability, diagnosable failure, and operator judgment.

The roadmap favors evidence-bearing useful-work progress over apparent autonomy.
Broader capability must carry the current discipline outward; it must not
replace it with opaque autonomy or an alternate control system.

## 7. Active Strategic Bottlenecks

- Governance strength must now convert into product usefulness without losing explicit control.
- Operator burden must be balanced against the value of transparent oversight.
- Anonymous flexibility must be reconciled with durable, case-scoped identity.
- The neutral core must remain distinct from consulting-specific design.
- Human-readable management views must remain consistent with machine structure.
- A useful dashboard must not become premature interface lock-in.
- trusted_local_unsandboxed is a trusted-worker posture, not comprehensive operating-system containment.
- Provider/model generalization remains unproven, and main.py remains a separate legacy migration concern.
- Repeated real-world user, value, burden, trust, privacy, and willingness-to-pay evidence is still absent.

## 8. Roadmap Phases

### Phase 1 — Foundation and Governance Core

Status: SUBSTANTIALLY_COMPLETE_AT_ALPHA_LEVEL

This phase establishes governance and role discipline, canonical bounded
execution, persistence and artifact lifecycle, deterministic verification,
human disposition, and an accepted exact provider-backed proof.

Exit condition: a coherent bounded alpha spine is proven and adopted.

### Phase 2 — Founder Visibility and Cockpit

Status: ACTIVE_ADVISORY_SIDECAR

This phase covers Founder Cockpit information architecture, read-only state
visualization, proof/caveat visibility, authority and freshness indicators, and
a fixture-backed prototype before live integration. The static design/prototype
exit condition is satisfied: the accepted six-unit hierarchy was reviewed in a

Static design/prototype exit condition: satisfied. This does not complete or
deploy a Cockpit, or establish accessibility, responsive, interaction,
independent-user, platform, live-data, or production proof.

### Phase 3 — Producer Adoption of Entry Preservation

Status: COMPLETED_AT_SOURCE_TEST_LEVEL

One existing in-memory case-packet producer adopts explicit
create/preserve/edit/replace/retire behavior for case-scoped source-material
and extracted-fact entries. Caller-provided identity remains explicit and
case-scoped; legacy anonymous behavior remains compatible; no semantic matching
or broad migration was added.

Entry condition: the pure preservation contract is ratified and durable.

Exit condition: one producer emits and preserves identified source/fact entries
under focused tests, without persistence or provenance overclaim. Satisfied at
source/test level by 4bcdc1bb43d2aab16b3db66131943a690a785786; this is not
persistence, provenance, workflow, product-wedge, or production proof.

### Phase 4 — Persistence Integration

Status: NOT_STARTED

This phase persists structured entries, retains legacy anonymous compatibility,
and prevents continuity bypass through whole-packet replacement without creating
an unnecessary global registry or history system.

This is the next technical phase following Phase 3, but it is not authorized by
this roadmap update.

Entry condition: producer behavior is proven.

Exit condition: save/load preserves explicit identities and transitions
deterministically, migration posture is explicit, and adoption is not silent.

### Phase 5 — Provenance Adoption

Status: DEFERRED

This phase introduces one canonical association set, forward and reverse views,
unlinked-entry visibility, and stale-link handling after replacement or
retirement. Association is not proof.

Entry condition: stable producer and persistence behavior exists.

Exit condition: provenance survives controlled edits and reloads, human-readable
views derive consistently, and durable link IDs are not required unless
separately proven.

## 9. Entry and Exit Gates

Roadmap progression requires the stated entry and exit conditions for each
phase, plus evidence appropriate to the affected layer. Management acceptance
does not replace source, test, runtime, or operator proof. Any phase crossing
that introduces mutation, provider/model execution, a dashboard control
surface, product-wedge selection, or production operations requires a separate
explicit authorization boundary.

## 10. Current Phase and Immediate Priorities

The project is operating between completed alpha foundation and the first
productization-oriented sequence. Immediate management priorities are:

1. complete CTO handoff and preserve continuity;
2. rank any future Founder Cockpit work separately against useful-work needs;
3. treat Phase 4 persistence integration as the next technical phase, subject to a separate authorization and ranking against useful-work needs;
4. avoid additional generic framework expansion without demonstrated need;
5. move toward one complete founder-legible dossier workflow; and
6. begin generating real proving-use-case evidence before product-wedge selection.

This ranking does not authorize any implementation track.

## 11. Deferred Decisions

## 12. Major Risks

Key risks are mistaking structural or deterministic proof for useful-work proof;
letting a consulting use case define the neutral core; locking an interface,
provider, model, or wedge before evidence exists; treating historical runtime
evidence as current availability; and allowing management synthesis to replace
technical or operational evidence.

## 13. Key Success Factors

Success depends on preserving authority boundaries, selecting narrow
evidence-bearing work, keeping the operator in honest contact with progress and
limits, using founder-legible views without new control semantics, and gathering
enough repeated real-world evidence to support a founder decision about product
direction.
