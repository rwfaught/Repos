# ORCHESTRATOR_METHOD_2_0_INTEGRATION.md

## Orchestrator Method 2.0 Integration: Make the Governance Upgrade Live

---

## Goal

Integrate the ratified `ORCHESTRATOR_METHOD_2_0_ALIGNMENT.md` rules into the live governance stack so that the method upgrade becomes operational rather than merely adjacent.

This is a **governance/alignment artifact**.

It is not:
- a feature phase
- a product-strategy artifact
- a broad documentation rewrite
- a code implementation task

Its purpose is narrower:

to incorporate the ratified 2.0 method rules into the actual governing documents with the smallest bounded edit surface necessary.

---

## Why This Artifact Exists

`ORCHESTRATOR_METHOD_2_0_ALIGNMENT.md` has been ratified.

That means the project has already accepted, in substance, that the old method layer is no longer explicit enough for the current project stage.

If the upgrade remains only as a saved side artifact, the project risks repeating the very failure mode the upgrade was meant to solve:
- ratified guidance exists
- but live governance behavior remains under-specified
- so conversation and ranking drift back toward prior habits

This artifact exists to prevent that.

The method upgrade should become part of the active governance stack before we use it to re-rank forward work.

---

## Scope

This artifact should make the 2.0 method rules live through small, explicit updates to the minimum set of governance docs.

Primary target:
- `docs/ORCHESTRATOR_METHOD.md`

Secondary targets only if clearly necessary for active-stack coherence:
- `docs/STARTUP_BRIEF.md`
- `docs/PROJECT_CONTEXT.md`
- `docs/ACTION_LOG.md`

Choose the smallest edit set that makes the upgraded method operational.

---

## This Artifact Is NOT About

This artifact is NOT about:

- replacing Ro
- rewriting `ORCHESTRATOR_METHOD.md` from scratch
- changing project strategy
- reopening proving-ground selection
- reorganizing the whole docs tree
- adding code behavior
- broadening into response-style experimentation
- creating a new bureaucracy layer for its own sake

This is a bounded integration step only.

---

## Required Integration Outcomes

### 1. Method integration
`ORCHESTRATOR_METHOD.md` should explicitly absorb the ratified 2.0 method rules, including at minimum:

- active layer rule
- decision-unlocked rule
- re-rank trigger rule
- active decision membrane rule
- document-authority classification rule
- design-stack stop rule
- response-protocol externalization rule
- repo-truth supremacy rule
- live constraint summary rule
- new-artifact admission rule

These do not need to be copied verbatim if concise integration is cleaner, but they must become live governing method.

### 2. Startup alignment if needed
If startup/re-entry clarity would remain incomplete without it, `STARTUP_BRIEF.md` may get one narrow addition acknowledging the method 2.0 upgrade as active restart/working discipline.

Only do this if needed.

### 3. Context alignment if needed
If `PROJECT_CONTEXT.md` needs one short note to preserve authority ordering or active-layer clarity, add it narrowly.

Only do this if needed.

### 4. Action log trace
`ACTION_LOG.md` should record that the method 2.0 alignment was integrated.

Keep that entry concise.

---

## Design Rules

1. Keep the integration small and factual.
2. Prefer incorporation over duplication.
3. Preserve the purpose of each touched document.
4. Do not turn the method doc into a bloated theory essay.
5. Make the upgrade operational, not rhetorical.
6. Do not touch non-governance docs.
7. Do not proceed into re-ranking from this artifact itself.
