# Founder Comprehension Snapshot Template

Boundary class: hinge artifact / founder visibility.

Status: reusable template.

Marker: `FOUNDER_COMPREHENSION_SNAPSHOT_TEMPLATE_REGISTERED=DOCS_ONLY`

## Purpose

A Founder Comprehension Snapshot is a short, source-grounded cockpit reading. It
is produced at project hinges so Roger can understand the project without
having to reverse-engineer the repo from phase machinery.

It is not a weekly memo. It is triggered by project gravity.

## Trigger Checklist

Produce this snapshot when any of the following is true:

- a Codex/worker campaign completed
- a multi-phase campaign completed
- a major handoff is being prepared
- Roger reports black-box concern
- implementation is about to resume after a pause
- a product wedge is being selected, changed, or silently reinforced
- another governance/readback phase is being ranked instead of visible product
  proof
- prior proof is being used to justify a broader claim than it actually proves

## Snapshot

### 1. Cockpit Summary

Where are we right now?

What can the system actually do?

What is mostly governance/readback rather than product capability?

What product direction is being implied?

What needs Roger's explicit decision before implementation resumes?

### 2. Accepted Facts

List directly inspected facts only.

Include:

- source inspected
- files inspected
- latest phase or checkpoint
- current boundary
- latest accepted markers if relevant
- source/capsule/git caveats

Do not include inference here.

### 3. Current Capability Reality

Classify each capability as one of:

- proven by deterministic tests
- operator-smoked
- runtime/provider-proven in a narrow way
- docs-only
- inferred but not proven
- not present / not found

For each capability, also state what it does not prove.

### 4. Recent Changes

What changed since the last founder-visible checkpoint?

Separate:

- new executable behavior
- new docs/control/governance
- deterministic readbacks
- tests around existing semantics
- product design
- runtime/provider proof
- source/capsule/export proof
- repair/handoff/posture work

### 5. Project Body vs Control Spine

What part of the project is becoming more capable?

What part is becoming heavier?

Is the control spine helping the product body move, or is it becoming the main
thing being built?

### 6. Product Direction Implication

What product direction is being implied by the latest work?

Is that direction:

- founder-ratified
- historically documented but currently reopened
- inferred from implementation momentum
- undecided

### 7. Strategic Decision Surface

What decision is being silently implied?

What options are live?

What is the recommended bridge if a final decision is not ready?

What would count as Roger ratification?

### 8. Non-Proofs

List what the latest proof does not prove.

Common examples:

- semantic correctness
- production readiness
- autonomous capability
- local model competence
- live provider/model behavior
- end-to-end product usefulness
- user-facing workflow readiness
- founder ratification

### 9. Next Bounded Move

State the recommended next move.

Include:

- boundary name
- purpose
- allowed operations
- exclusions
- expected proof
- whether mutation is authorized
- why this move outranks alternatives

### 10. Response Metadata

Include:

- active boundary
- mode
- changed files
- mutation status
- source/capsule status
- open threads
- next boundary
- proof/validation status
- founder ratification status

## Style Standard

Use white-paper clarity:

- concise executive summary first
- plain English definitions
- enough detail to preserve reality
- no ritual density unless necessary
- no unsupported confidence
- no broad claims from narrow proof

The output should make Roger more dangerous as Founder/Operator, not more
dependent on AI explanation.
