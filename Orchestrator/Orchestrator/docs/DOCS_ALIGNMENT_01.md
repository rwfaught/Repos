# DOCS_ALIGNMENT_01.md

## Docs Alignment 01: Vision-Anchor Integration into Governance Stack

---

## Goal

Integrate `PROJECT_VISION.md` into the governance and orientation stack through small, explicit documentation updates that preserve each document’s existing job while making the new vision artifact part of the project’s ranking and startup logic.

This is a **docs/control-surface alignment artifact**.

It is not a feature phase.
It is not a fix phase.
It does not authorize product behavior changes.

Its purpose is to make the approved vision artifact legible and operative in the documents that govern ranking, startup orientation, and project framing.

---

## Why This Artifact Exists

`PROJECT_VISION.md` has now been ratified as the project’s constitutional direction.

That creates a new top-level anchor in the documentation stack.

If the rest of the governance/orientation docs do not reference it, the repo will carry the vision artifact as an isolated statement rather than as a live ranking and startup guide.

This artifact closes that gap in a bounded way.

---

## Scope

Add narrow references or small alignment passages to the minimum set of documents that should explicitly inherit the new vision anchor.

The preferred target set is:

- `docs/ORCHESTRATOR_METHOD.md`
- `docs/STARTUP_BRIEF.md`
- `docs/PROJECT_CONTEXT.md`
- `docs/ORCHESTRATOR_HANDOFF_STATUS_OUTLINE.md` only if a brief reference is clearly useful

Choose the smallest set of edits that makes the vision artifact part of the active governance stack.

---

## This Artifact Is NOT About

- rewriting the project’s philosophy documents broadly
- duplicating the full contents of `PROJECT_VISION.md` elsewhere
- changing the present-tense success bar in `CURRENT_SUCCESS_CRITERION.md`
- changing phase ordering
- changing fix handling
- adding product features
- adding intake/triage specification
- adding new roadmap commitments
- rewriting handoff/history prose beyond narrow alignment

The goal is alignment, not diffusion.

---

## Required Alignment Outcomes

### 1. Method-level alignment
`ORCHESTRATOR_METHOD.md` should clearly reflect that future ranking should remain answerable not only to the current success criterion, but also to the ratified project vision.

This should preserve the existing distinction:
- `CURRENT_SUCCESS_CRITERION.md` = present-tense product bar
- `PROJECT_VISION.md` = long-range constitutional direction

Do NOT collapse these into one concept.

### 2. Startup-level alignment
`STARTUP_BRIEF.md` should instruct the orchestrator to consider `PROJECT_VISION.md` as part of startup orientation when relevant.

This should be done narrowly.
Do NOT overburden the startup script.

### 3. Context-level alignment
`PROJECT_CONTEXT.md` should acknowledge the ratified vision artifact and preserve the present/future distinction:
- current software-first bounded execution reality
- future direction toward broader honest-contact orchestration without overclaiming current intake capability

### 4. Optional handoff-level alignment
If useful and minimal, `ORCHESTRATOR_HANDOFF_STATUS_OUTLINE.md` may mention that `PROJECT_VISION.md` now exists as the project’s vision anchor.

Only do this if it materially improves future handoffs.

---

## Design Rules

1. Keep edits small.
2. Prefer short references over long restatements.
3. Do not copy large passages out of `PROJECT_VISION.md`.
4. Preserve each document’s current purpose.
5. Do not let the vision artifact blur present capability with intended direction.
6. Keep the stack layered and intelligible.

---

## Source Of Truth

For long-range direction:
- `docs/PROJECT_VISION.md`

For present-tense product truth:
- `docs/CURRENT_SUCCESS_CRITERION.md`

For orchestration/ranking behavior:
- `docs/ORCHESTRATOR_METHOD.md`

For startup orientation:
- `docs/STARTUP_BRIEF.md`

This artifact should strengthen those distinctions, not weaken them.

---

## Validation Requirements

Confirm that after the edits:

1. `PROJECT_VISION.md` is referenced where needed in the governance/orientation stack
2. no document now overclaims current intake capability
3. `CURRENT_SUCCESS_CRITERION.md` remains distinct from vision
4. the updated docs remain internally coherent and non-duplicative
5. no unnecessary broad rewrites were introduced
