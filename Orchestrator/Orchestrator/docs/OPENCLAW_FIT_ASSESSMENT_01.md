# OPENCLAW_FIT_ASSESSMENT_01.md

## Purpose

Assess OpenClaw fit against current Orchestrator strategy and boundary discipline.

## Classification

OpenClaw is currently:
- a strategic boundary check
- a possible future adapter / access-runtime shell
- a distraction for now
- not the project architecture
- not something Orchestrator should clone

## Layer Distinction

- OpenClaw-like systems are reach/action layers.
- Orchestrator is the bounded-trust state and authorization layer.

OpenClaw-like reach/action is valuable, but it does not replace kernel responsibilities.

## Required Boundary

- Orchestrator should not rebuild generic assistant/channel infrastructure unless forced.
- OpenClaw may later deliver commands to Orchestrator.
- Orchestrator must decide whether commands are authorized.
- OpenClaw must not own Orchestrator state transitions.
- Orchestrator must remain the bounded-trust kernel.
- Broad assistant/gateway work should not displace the golden-path casework proof.

## Provisional Decision

- OpenClaw is a future adapter.
- OpenClaw is a distraction for now.
- OpenClaw is useful later, after the bounded casework kernel proves value.
- No immediate OpenClaw integration is authorized.
