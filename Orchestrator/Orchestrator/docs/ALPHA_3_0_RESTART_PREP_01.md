## Document 1 — OPENCLAW_FIT_ASSESSMENT_01.md

### Purpose

Classify OpenClaw’s relationship to Orchestrator.

This document must answer whether OpenClaw is:

- competitor
- runtime/access layer
- future adapter
- distraction
- scope boundary
- integration candidate

The document must make clear that OpenClaw is strategically important but must not become the gravitational center of Orchestrator.

### Required claims

The document must state:

- OpenClaw-like reach/action is valuable.
- Orchestrator should not rebuild generic assistant/channel infrastructure unless forced.
- OpenClaw should be treated as a strategic boundary check.
- OpenClaw may become a future adapter or access/runtime shell.
- OpenClaw must not own Orchestrator’s state transitions.
- Orchestrator must remain the bounded-trust kernel.
- OpenClaw/channel surfaces may deliver commands.
- Orchestrator must decide whether those commands are authorized and what state changes are allowed.
- The project should avoid becoming an OpenClaw clone.
- The project should avoid broad assistant/gateway work before the golden-path casework proof.

### Required distinction

The document must preserve this distinction:

- OpenClaw: reach/action layer.
- Orchestrator: bounded-trust state and authorization layer.

Acceptable metaphor:

- OpenClaw can be mouth, ears, hands, channels, and muscles.
- Orchestrator must remain spine, ledger, brakes, case memory, and authorization membrane.

### Required decision

The document must end with a clear provisional decision:

- OpenClaw is a future adapter.
- OpenClaw is a distraction for now.
- OpenClaw is useful later, after the bounded casework kernel proves value.

It must not recommend immediate OpenClaw integration.

## Document 2 — OWNER_AUTHORED_SYSTEM_PRINCIPLE.md

### Purpose

Encode the “MY SYSTEM / THEIR SYSTEM” principle.

This document must say that Orchestrator is allowed to begin as the operator’s personally sovereign system, but must preserve architectural seams so other operators can eventually make it their own.

### Required claims

The document must state:

- Personal does not mean hardcoded.
- Portable does not mean generic.
- Customizable does not mean loose.
- Local-first does not mean isolated from other people’s use cases.
- Orchestrator should not merely be Roger’s workflow.
- Orchestrator should become a framework for owner-authored workflows.
- The system should be “my system” at point of use and “their system” at point of adoption.

### Required architectural implications

The document must identify future seams such as:

- local policy
- packet templates
- approval thresholds
- model preferences
- transport adapters
- workflow definitions
- operator-specific defaults
- privacy posture
- escalation rules
- case/work packet types

### Required non-authorizations

The document must explicitly not authorize:

- customization framework
- plugin marketplace
- multi-user support
- public template ecosystem
- UI theme/config system
- broad abstraction phase
- generic workflow engine

This is a principle document, not a build roadmap.
